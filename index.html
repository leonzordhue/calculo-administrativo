Criado por Maria Lúcia Carvalho Grana
Versão base do sistema. SPA monolítica (HTML/CSS/JS). Áreas: SEDUC, SEMED, SES, SEMSA, PMAM, PCAM. Motor de cálculo de diferenças salariais com SELIC (EC 113/2021), 13º proporcional, 1/3 férias, integração Google Sheets via Apps Script, live preview de formulário, stepper 3 passos.
CORRIGIDO: identificador alertaPrescrição (com cedilha) renomeado para alertaPrescricao. Causa: caracteres especiais em identificadores JS funcionam em motores modernos mas quebram em ambientes restritos, minificadores e copiar/colar manual. Evitar: nunca usar acentos ou cedilha em nomes de variáveis/propriedades JS. CORRIGIDO: areaCalc atribuía 'seduc_admin'/'semed_admin' (lowercase) enquanto REGRAS usa 'SEDUC_ADMIN'/'SEMED_ADMIN' (uppercase). Apesar de existir um .toUpperCase() posterior, a inconsistência era frágil. Normalizado para uppercase diretamente na atribuição. CORRIGIDO: legalMap e areaLabel em renderizarResultados não tinham chaves para SEDUC_ADMIN e SEMED_ADMIN. Resultava em fundamentação jurídica genérica (fallback silencioso) para cálculos administrativos. Adicionadas entradas com base legal correta de cada PCCR. CORRIGIDO: numProgressoesAtraso em gerarDemonstrativoSES podia ser negativo quando classeDevida era anterior a classeAtual. Envolvido em Math.max(0, ...) para garantir valor mínimo zero. CORRIGIDO: campo 's-categoria' ausente em allIds de bindPreviewListeners. Mudança de categoria não acionava atualização do live preview. Adicionados 's-categoria' e 'm-categoria' à lista. ATENÇÃO: taxas SELIC de 2025-04 em diante podem divergir do BCB. Conferir mensalmente em bcb.gov.br. Adicionado comentário no código.
CORRIGIDO (CRÍTICO): gerarDemonstrativoSeguranca calculava postoDevido na direção errada. Object.keys(cfg.postos/cargos) está ordenado do posto MAIOR (idx 0 = Coronel/Delegado Especial) para o MENOR (idx N = Aluno/Agente4). Promoção = REDUÇÃO de índice. O código usava Math.min(idxInicial + progsDevidas, length-1), movendo do maior para o menor posto (rebaixamento). Para PM com posse em 1989: dezenas de promoções calculadas → posto fictício abaixo do atual → devItem.sal menor → diff ≤ 0 → tabela vazia. Corrigido para Math.max(idxInicial - progsDevidas, 0). Evitar: sempre verificar a semântica de ordenação do objeto antes de somar/subtrair índices de progressão. CORRIGIDO: mesmo padrão do SES — numProgressoesAtraso em SEMSA sem Math.max(0, ...). Corrigido. Padrão: sempre proteger contagens de progressão com Math.max(0, ...). CORRIGIDO: bloqueio de data fim futura só existia em _coletarEduc. SES, SEMSA, PMAM e PCAM permitiam calcular períodos futuros. Adicionada validação (endDate > hoje → throw Error) em todos. CORRIGIDO: alerta de prescrição quinquenal (Dec. 20.910/1932) só era emitido para SEDUC/SEMED. SES, SEMSA, PMAM, PCAM não alertavam. Adicionado em todas as funções _coletar*. REMOVIDO: variável global AREA_ATUAL declarada e nunca usada. Dead code — removido para reduzir superfície de confusão. CORRIGIDO: addInfo em renderizarResultados usava 'area' (parâmetro desestruturado de res) em vez de 'areaKey' para SES/PMAM/PCAM. Normalizado para areaKey em todos os branches. CORRIGIDO: tagIco e result-tags não cobriam variantes admin/apoio. Renderizava ícone '📋' genérico e classe CSS undefined para SEDUC_ADMIN, SEMED_ADMIN, SEDUC_APOIO_*. Expandido. CORRIGIDO: payload enviado ao Google Sheets não incluía totalCorrigido (SELIC) nem fatorSelic/diffCorrigida por linha. Planilha recebia apenas valor nominal. Adicionados todos os campos de correção. CORRIGIDO: células de vencRecebido/vencDevido na tabela usavam 'r.vencRecebido ?' (falsy check). Um valor R$ 0,00 exibiria '—'. Trocado para '!= null' (null check correto).
CORRIGIDO (CRÍTICO): motor de progressão da PCAM usava lista global de cargos como escada única (Delegados → Agentes → Peritos em sequência). Um agente com muitos anos de serviço "progride" além de agente_especial e cruza para a carreira de delegado — juridicamente impossível (carreiras distintas, concursos distintos). Solução: adicionado objeto 'trilhas' ao REGRAS.PCAM com listas separadas por carreira (delegados / agentes / peritos). Mesmo padrão aplicado ao REGRAS.PMAM (pracas / oficiais_qopm / qoapm). Criadas funções getTrilhaProgresso() e calcPostoDevido() que respeitam o teto de cada trilha. calcProgressoesSeguranca() removida (dead code). ADICIONADO: validação explícita em _coletarPCAM e _coletarPMAM. Se postoAtual e postoInicial pertencerem a trilhas distintas, lança erro descritivo orientando o operador, em vez de calcular silenciosamente valores errados.
Inserção do sistema de log de versões (este bloco). Todas as correções REV-1 a REV-4 documentadas retroativamente. window._VERSION_LOG e window._VICIOS_CONHECIDOS disponíveis via console (F12) para auditoria e manutenção.
CORRIGIDO: "Nenhuma diferença encontrada" para PMAM/PCAM quando o operador não sabe/não informa a data de ingresso original, ou quando o PM já foi promovido e quer apenas o retroativo do período de atraso. Causa raiz: motor calculava postoDevido exclusivamente via posseDate + N×interstício. Se posseDate for recente ou errada, progsAteMes = 0 para todo o período → postoDevidoMes = postoAtual → todas as linhas puladas → tabela vazia sem nenhum erro. Solução: campo opcional "Posto que Deveria Estar" (pm-posto-devida / pc-quadro-devida). Quando preenchido, o motor usa esse valor como postoDevidoMes fixo para todos os meses, eliminando a dependência da data de ingresso. Validações: trilha cruzada + verificação de que devida > atual na hierarquia. Novos campos no formulário PMAM e PCAM: pm-posto-devida e pc-quadro-devida (selects opcionais com hint de aviso). Adicionados ao allIds do live preview e ao PREVIEW_FIELDS. Parâmetro postoDevidaFixo propagado por gerarDemonstrativoSeguranca, loop mensal e loop de extras (13º/férias).
CORRIGIDO: cálculo iniciava somente após o primeiro interstício medido desde posseDate (data de ingresso na corporação). Para PM promovido há anos ao posto atual (ex: sgt2 em 2018 mas ingresso em 2010), o motor calculava 36 meses a partir de 2010, gerando primeira diferença em 2013 — mas se o retroativo foi pedido a partir de 2021, os meses de 2021 a 2022 eram descartados silenciosamente. Caso real: Johnny Bronson pediu desde 2021, apareceu só 2023. Solução: campo "Data da Última Promoção ao Posto Atual" (pm-data-promocao / pc-data-promocao). Quando preenchido, o motor usa essa data como dataReferenciaIntersticio em vez de posseDate para medir o interstício. Adicionado alerta visual quando linhas foram descartadas antes do startDate. Vício V10 documentado.
CORRIGIDO: erro "Vigência não encontrada" para todos os meses entre startDate e dataReferenciaIntersticio quando dataRef > startDate. Causa: diffMonths(dataRef, cursor) retorna negativo quando cursor está antes de dataRef. Math.floor(-5/36) = -1 → calcPostoDevido recebia nProgs=-1 → trilha[idxT-1] = trilha[-1] = undefined → calcVencimentoSeguranca(area, undefined, ...) → cfg.postos[undefined] = undefined → retorna null → erro "Vigência não encontrada" para cada mês até dataRef. Fixes: (1) calcPostoDevido protegido — nProgs menor ou igual a 0 retorna postoInicial imediatamente; (2) loop mensal pula meses onde mesesAteAgora < 0 (cursor antes de dataRef); (3) loop de extras (13º/férias) com mesma proteção.
CORRIGIDO: mensagem "Nenhuma diferença encontrada" era genérica e não orientava o operador. Para PMAM e PCAM, o sistema agora detecta a causa provável e exibe orientação específica: se nenhum campo auxiliar foi preenchido (pm-posto-devida / pm-data-promocao), orienta o operador a usar a Opção 1 (posto devida) ou Opção 2 (data da promoção). Se a data de promoção foi preenchida mas o interstício não completou, informa isso explicitamente com o valor informado e sugere alternativa.
CORRIGIDO: "Nenhuma diferença" quando servidor já foi promovido e quer retroativo do período de atraso. Sistema agora detecta automaticamente: se postoDevido calculado === postoAtual E postoInicial está abaixo na trilha, ativa modoAutoDetectado — trata postoInicial como "recebido" e postoAtual como "devida" para todos os meses do período. Elimina necessidade de campo manual "Posto que Deveria Estar" na maioria dos casos. Banner informativo exibido na tela de resultados quando auto-detecção foi ativada.
Fix parcial de inversão automática. Falha quando progsAteMes >= 1 retorna postoDevidoMes = postoAtual — inversão não dispara. VICIO_EVITAR: inversão dinâmica por comparação de índice NÃO é suficiente quando postoDevidoMes pode igualar postoAtual. Use o branch autoDetectado com valores fixos.
CORRIGIDO: "só 2023 pra cá" mesmo sem pm-data-promocao. Causa raiz: loop usava inversão dinâmica que falha quando progsAteMes≥1 retorna postoDevidoMes=postoAtual → inversão não dispara → ambos sgt1 → skip. Fix: branch autoDetectado bypassa completamente o cálculo de progs. Quando postoInicial menor que postoAtual na trilha: recebido=postoInicial, devida=postoAtual para TODOS os meses do período. VICIO_EVITAR: inversão dinâmica é insuficiente quando postoDevidoMes pode igualar postoAtual em algum mês. Sempre usar branch autoDetectado com postoRecebidoMes/postoDevidoMes fixos quando retroativo de promoção. Sistema de log migrado para formato estruturado com data-id em cada span, fixes como objetos {id,desc} no _VERSION_LOG JS, console aprimorado.
Fix com campo manual s-ref-max / m-ref-max. Descartado — contrário ao princípio de automação. Substituído por cap automático na REV-14. VICIO_EVITAR: não criar campos manuais para informações que o sistema pode calcular com os dados já disponíveis.
CORRIGIDO: progressões além do devido sem campo manual. Cap automático: progsMaximas = 1 + floor((endDate - startDate) / intersticio). O "1" representa a progressão já devida no início do retroativo. As demais representam quantas cabem dentro do período. Campos manuais s-ref-max / m-ref-max removidos. Verificação: A1, MAR/2023→MAR/2026, intersticio=24 → 1 + floor(36/24) = 2 progs → teto C1 ✓
CORRIGIDO: formulário SES usava Classe I–VIII (8 posições) em vez da notação oficial da Lei 3.469/2009 (A1–D4 para Sup/Méd, E1–H4 para Aux I). Reestruturação: REGRAS.SES.multRefs expandido para 32 chaves (16 SM + 16 AI), cada uma com +2% por posição. Trilha detectada automaticamente pelo prefixo. Motor usa classeComCap() com cap automático por período (herda padrão REV-14). Função atualizarClasseInicialSES() filtra select de ingresso pela mesma trilha. Caso de uso direto: "está na E1, deveria estar na F2" — selecionar E1 em "Classe no Ingresso" e preencher período retroativo.
CORRIGIDO: 13º e 1/3 férias usavam classe diferente da mostrada nos meses do ano. Ex: DEZ/2021 mostrava E4 mas 13º 2021 mostrava F1. Causa raiz: _appendExtras e loop de extras do Educ recalculavam refDevidaDez via diffMonths(posseDate, dezembro1) / 24, ignorando o cap automático do loop mensal → saltavam para classe além do cap. Fix: ambos os geradores agora buscam a refDevida da ÚLTIMA row normal do ano nas rows já calculadas, garantindo consistência total entre linhas mensais e linhas de 13º/férias. VICIO_EVITAR: nunca recalcular refDevida para 13º/férias a partir de posseDate — usar sempre a última row do ano (ultimaRow.refDevida).
VÍCIO V01 — DIREÇÃO DE PROGRESSÃO EM OBJETOS ORDENADOS: Object.keys() de postos/cargos está ordenado do MAIOR para o MENOR (Coronel=idx0, Aluno=idxN). Progressão (promoção) = SUBTRAÇÃO de índice (idxInicial - nProgs). NUNCA somar. Sempre usar calcPostoDevido() que abstrai essa lógica. VÍCIO V02 — CRUZAMENTO DE TRILHAS DE CARREIRA: PCAM e PMAM têm carreiras independentes com progressões restritas à trilha de ingresso. NUNCA tratar a lista de cargos como escada linear única. Sempre usar getTrilhaProgresso() + calcPostoDevido(). VÍCIO V03 — CONTAGEM DE PROGRESSÕES SEM PROTEÇÃO: numProgressoesAtraso pode ser negativo se a classe/posto atual for maior que o calculado (dados inconsistentes ou cenário edge-case). SEMPRE proteger com Math.max(0, ...). VÍCIO V04 — VALIDAÇÕES ASSIMETRICAS POR ÁREA: Regras de negócio (data futura, prescrição quinquenal) devem ser aplicadas em TODAS as funções _coletar* (educ, SES, SEMSA, PMAM, PCAM). Ao adicionar nova validação, revisar todas as 5+ funções. VÍCIO V05 — CHAVES DE ÁREA INCONSISTENTES: res.area pode ser 'seduc', 'SEDUC_ADMIN', 'SEDUC_APOIO_SUPERIOR' etc. Nunca comparar com strings fixas sem cobrir TODAS as variantes. Usar areaKey = res.area || area e cobrir todos os casos nos maps. VÍCIO V06 — FALSY CHECK EM VALORES MONETÁRIOS: 'r.valor ?' falha para R$ 0,00 (falsy). SEMPRE usar '!= null' ou 'r.valor !== undefined' ao checar existência de valores numéricos. VÍCIO V07 — IDENTIFICADORES COM CARACTERES ESPECIAIS: Nunca usar acentos (ç, ã, é) em nomes de variáveis, propriedades ou chaves de objeto JS. Funciona em motores modernos mas quebra em minificadores, ambientes restritos e cópia manual. VÍCIO V08 — DEAD CODE ACUMULADO: Variáveis globais declaradas e nunca usadas (ex: AREA_ATUAL) e funções inutilizadas (ex: calcProgressoesSeguranca antes da remoção) confundem manutenção. Remover ao identificar. VÍCIO V09 — DEPENDÊNCIA EXCLUSIVA DE posseDate NO MOTOR PMAM/PCAM: O motor calcula postoDevido via posseDate + N×interstício. Se o operador informar data errada (ex: data da última promoção em vez do ingresso original), progsAteMes = 0 → tabela vazia sem erro. SEMPRE oferecer campo postoDevidaFixo como override. Quando preenchido, usar esse valor diretamente (modoFixo = true), ignorando posseDate. VÍCIO V10 — CONFUSÃO ENTRE posseDate E dataReferenciaIntersticio: posseDate = data de ingresso na corporação (ex: 2010). dataReferenciaIntersticio = data em que o militar foi promovido ao posto ATUAL — que é o ponto de partida real do interstício (ex: 2018). Usar posseDate para medir interstício faz o motor pular meses válidos. SEMPRE oferecer campo pm-data-promocao / pc-data-promocao. Quando preenchido, usar parseDate desse campo como dataRef, não posseDate. VÍCIO V11 — diffMonths NEGATIVO DESTRÓI ARRAY INDEX: Se cursor está antes de dataRef, diffMonths(dataRef, cursor) é negativo. Math.floor(negativo/36) ainda é negativo. calcPostoDevido recebe nProgs negativo → trilha[idxT + negativo] = trilha[-1] = undefined → passa undefined para calcVencimentoSeguranca → retorna null → "Vigência não encontrada" para cada mês afetado. SEMPRE proteger: (1) calcPostoDevido com guard nProgs <= 0 → return postoInicial; (2) loop mensal com if (mesesAteAgora < 0) continue; (3) loop extras com if (mesesDez < 0) return.
